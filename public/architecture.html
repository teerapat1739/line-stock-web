⚙️ หัวใจของระบบ — Postgres Function ที่ป้องกันการชนกัน

Edge Function เรียก rpc('apply_entry', ...) — ทุกอย่างเกิดใน transaction เดียว ถ้ามีคนอื่นแก้รายการเดียวกันก่อน version จะไม่ตรง → ROLLBACK และตอบ collision

-- เรียกเมื่อบันทึกใหม่ หรือแก้ของเดิม
create or replace function apply_entry(
  p_item_id  uuid,
  p_action   text,
  p_qty      numeric,
  p_unit     text,
  p_user_id  text,
  p_edit_of  uuid default null,    -- ถ้าเป็นการแก้
  p_expected_version int default null
) returns jsonb language plpgsql as $$
declare
  v_delta  numeric;
  v_after  numeric;
  v_entry  entries;
begin
  -- ① ถ้าเป็นการแก้ ล็อกแถวเดิมไว้ก่อน เช็ค version
  if p_edit_of is not null then
    select * into v_entry from entries
      where id = p_edit_of for update;
    if v_entry.version <> p_expected_version then
      return jsonb_build_object(
        'ok', false, 'collision', true,
        'message', 'รายการนี้เพิ่งถูกแก้/ลบโดยคนอื่น');
    end if;
    update entries set status='edited', version=version+1
      where id = p_edit_of;
    v_delta := signed_delta(v_entry.action, p_qty) - signed_delta(v_entry.action, v_entry.qty);
  else
    v_delta := signed_delta(p_action, p_qty);
  end if;

  -- ② อัปเดต current_qty ใน items (atomic)
  update items set current_qty = current_qty + v_delta
    where id = p_item_id
    returning current_qty into v_after;

  -- ③ append entry ใหม่ (audit row)
  insert into entries (item_id, action, qty, unit, by_user_id, ref_id, status)
    values (p_item_id, p_action, p_qty, p_unit, p_user_id, p_edit_of,
            case when p_edit_of is null then 'active' else 'active' end);

  return jsonb_build_object('ok', true, 'after', v_after);
end; $$;

📦 สิ่งที่ใช้ใน Supabase

  • Edge Functions เขียนด้วย TypeScript / Deno รับ webhook + เรียก LINE Reply API
  • Postgres เก็บ records, items, sessions ทุกอย่างใน DB เดียว
  • Realtime push การเปลี่ยนแปลงสด ๆ ไปที่ dashboard ผ่าน WebSocket
  • Database Functions (apply_entry) เป็นจุดเดียวที่อัปเดตยอด — ป้องกัน race condition
  • pg_cron ลบ session หมดอายุทุก 5 นาที
  • RLS dashboard อ่านได้ตามสิทธิ์ Edge Function เขียนด้วย service role
  • Storage เผื่อในอนาคต รับรูปบิล/ใบเสร็จ จาก LINE

⚔️ ทำไมหลายคนพร้อมกันไม่ตีกัน (บน Postgres)

  • SELECT ... FOR UPDATE ภายใน apply_entry — ล็อกแถว entry ที่กำลังแก้ คนอื่นต้องรอ
  • Optimistic version check — ถ้า version ที่ส่งมาไม่ตรงกับใน DB → return collision ทันที
  • Transaction — UPDATE items + INSERT entries อยู่ในจังหวะเดียว ถ้าพังจุดไหน rollback ทั้งหมด
  • Sessions แยกตาม userId — เก็บในตาราง sessions (PK = line_user_id) จึงไม่ปะปนกัน
  • Realtime ไม่กระทบ correctness — แค่ broadcast event ตามหลัง ไม่ใช่ source of truth

🛡 ความปลอดภัย

  • LINE signature ตรวจสอบใน Edge Function ทุก request
  • Service Role Key เก็บใน supabase secrets set ไม่ใส่ใน repo
  • RLS เปิดทุก table — anon key อ่านได้แค่ที่ policy อนุญาต
  • Group allowlist — บอททำงานเฉพาะ groupId ที่ลงทะเบียนใน locations.line_group_id
  • Webhook URL ใช้ Supabase domain เท่านั้น (HTTPS in)

🧱 ขั้นตอน deploy

  • สร้าง Supabase project (ฟรี) ได้ SUPABASE_URL + คีย์ 2 แบบ
  • รัน migration SQL — สร้าง 5 ตาราง + function apply_entry + RLS policies
  • สมัคร LINE Developers → สร้าง Messaging API channel ได้ CHANNEL_SECRET + CHANNEL_ACCESS_TOKEN
  • supabase functions deploy line-webhook — ได้ URL https://<proj>.supabase.co/functions/v1/line-webhook
  • เอา URL ไปใส่ใน Webhook URL ของ LINE channel → เปิด "Use webhook" → ปิด "Auto-reply"
  • เชิญบอทเข้ากลุ่ม → ส่ง "สวัสดี" ทดสอบ
  • (เลือก) deploy Next.js dashboard ลง Vercel — ใช้ anon key + RLS
